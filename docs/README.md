## Documentation Principles

### Single Source of Truth
- Each topic has ONE authoritative document
- No duplication across files
- Cross-reference via links, not copy-paste

### Organization
- **Brand** - Product identity and positioning (for users)
- **Architecture** - System design and structure (for developers)
- **Standards** - Quality and implementation requirements (for contributors)
- **Guides** - How-to documentation and workflows (for all)

### When to Update

| Document | Update When |
|----------|-------------|
| ARCHITECTURE.md | Package structure changes, dependency graph changes |
| COMPONENT_STANDARD.md | New component patterns, API design changes |
| ACCESSIBILITY.md | New a11y requirements, WCAG updates |
| TESTING_STRATEGY.md | Test infrastructure changes, coverage policy changes |
| DEVELOPMENT_RULES.md | Naming convention changes, new development patterns |
| CONTRIBUTING.md | PR workflow changes, commit convention changes |
| PRODUCT_BRAND.md | Brand strategy changes, positioning updates |

## Maintenance

Documentation is code. Keep it:
- **Accurate** - Verify against actual code
- **Current** - Update when implementation changes
- **Concise** - Remove outdated content
- **Linked** - Cross-reference related docs
- **Organized** - Use folder structure for clarity
